# Portcall

A lightweight SwiftUI menu bar utility for macOS that continuously monitors
listening TCP/UDP ports and shows which process owns each one, in real time.

The menu bar shows a live count of listening services (with a warning glyph when
any are exposed to the network). The dropdown lists each service with its
process, PID, protocol, bound address, CPU/memory, and an exposure indicator
(localhost-only vs. reachable from the network), plus search and quick actions
(Open in Browser, Reveal Binary in Finder, Copy, Terminate).

## Architecture

```
MenuBarExtra (UI)  ─▶  PortStore (@MainActor ObservableObject)  ─▶  PortScanner
   live count            services, timer, search, CPU deltas          libproc sockets
                                  │
                                  └▶  ProcessInfoCollector  (libproc task info + path)
```

- **`Libproc`** — thin Swift wrappers over the libproc C API (`proc_listpids`,
  `PROC_PIDLISTFDS`, `PROC_PIDFDSOCKETINFO`, `PROC_PIDTASKINFO`, `proc_pidpath`).
  No subprocesses.
- **`PortScanner`** — walks every PID's socket fds via libproc, keeping TCP
  sockets in LISTEN state and bound/unconnected UDP sockets, normalized into
  `PortEntry` values.
- **`ProcessInfoCollector`** — per-PID raw `ProcessSample` (cumulative CPU mach
  ticks, resident memory, true binary path via `proc_pidpath`, robust to argv[0]
  rewriting like ssh).
- **`PortStore`** — owns the 5s scan timer, runs scans off the main actor,
  derives **live %CPU** from successive samples (mach-ticks delta ÷ wall-clock,
  converted via `mach_timebase_info`), and publishes grouped `ServiceRow`s plus
  `[PID: ProcessStats]` and container maps.
- **`MenuContentView`** — the dropdown: header with counts, search, the service
  list, and a footer.

## Build & run

Two interchangeable build systems share the same `Sources/Portcall`:

**Xcode (for debugging in the IDE)** — project is generated by
[XcodeGen](https://github.com/yonaskolb/XcodeGen) from `project.yml`:

```sh
xcodegen generate       # regenerate after adding/removing source files
open Portcall.xcodeproj # then ⌘R to run
```

The app target is unsandboxed with a real bundle identifier
(`com.local.portcall`), so it runs cleanly without the bundle-less-binary
console warnings. **Re-run `xcodegen generate` whenever you add or remove
files** — sources are globbed at generation time.

**SwiftPM (for terminal builds / CI / headless scans):**

```sh
swift build && swift run            # debug run
swift run Portcall --scan           # print parsed ports and exit
./Scripts/build-app.sh              # release → build/Portcall.app
open build/Portcall.app             # menu-bar app (no Dock icon)
```

> Do **not** enable App Sandbox — a sandboxed build cannot read other processes'
> fds via libproc or call `kill()`. The Xcode target ships with no entitlements
> file on purpose.

## Notes & limitations

- Run as a normal user, libproc can't list file descriptors of processes owned
  by other users (root), so some system listeners may be missing. A privileged
  scan path is future work. (Same practical limit as a non-root `lsof`.)
- UDP has no "LISTEN" state; bound, unconnected UDP sockets are treated as
  listening; connected flows and unbound (port 0) sockets are filtered out.
- Live %CPU needs two samples, so a process's CPU reads `0.0%` for the first
  scan after it (or the app) starts, then updates each interval.

## Roadmap

- **Phase 1** ✅ — menu-bar app, lsof scanner, searchable list, exposure
  indicator, copy/terminate.
- **Phase 2** ✅ — per-process CPU/memory (`ps`) and true binary path
  (`proc_pidpath`, robust to argv[0] rewriting like ssh); quick actions:
  Reveal Binary in Finder, Open in Browser (http/https heuristic, TCP-only),
  Copy Executable Path.
- **Phase 3** ✅ — Docker/Colima container correlation (optional; only active
  when a container runtime is present). Matches host ports to `docker ps`
  published ports, which also resolves the Colima ssh/VM port-forward case.
  Shows a container badge per matched row + header count; `docker ps` runs with
  a 2s timeout so an unreachable daemon never stalls the scan. Container polling
  is **decoupled** from the 5s port scan — `docker ps` runs at most every 20s
  (or on manual refresh), since the subprocess (~15ms) dwarfs the ~1ms libproc
  scan.
- **Phase 4** ✅ — scanner rewritten on libproc (no `lsof`/`ps` subprocess);
  per-process CPU/memory via `proc_pidinfo(PROC_PIDTASKINFO)`; **true live CPU**
  via mach-ticks delta over wall-clock (timebase-corrected — `pti_total_*` are
  mach units, not nanoseconds).
- **Dedupe polish** ✅ — `ServiceRow` collapses IPv4+IPv6 sockets of the same
  process + protocol + port + exposure into one dual-stack row (tagged `v4·v6`),
  so counts read as services rather than sockets. `PortScanner` still emits raw
  per-socket data; grouping is presentation-only.
- **Settings** ✅ — footer has gear + power (quit) icon buttons. The gear opens
  the "Portcall Settings" window (a `Settings` scene) with: **Launch at login**
  (real OS login item via `SMAppService.mainApp`), **refresh interval**, **CPU
  mode** (Activity-Monitor vs. percent-of-all-cores), and **hide system
  services** (filters Apple daemons by executable-path prefix; keeps user tools
  and ssh-forwarded container ports). Preferences persist in `UserDefaults`;
  `PortStore` reschedules the scan timer when the interval changes.

> Launch-at-login registers the *currently running* bundle. For a stable login
> item, run the app from `/Applications` (a DerivedData build registers that
> transient path).
